# LuxLedger Demo Release Checklist — OpenAPI 0.2.0

Use this checklist for the LL-63 documentation release. Record evidence (CI URL or command output) in the PR.

- [ ] Demo package version: `luxledger-demo-api` `0.1.0`.
- [ ] `@luxledger/core`: installed/locked `0.1.1`.
- [ ] `@luxledger/http`: installed/locked `0.1.1`.
- [ ] `@luxledger/fastify-routes`: installed/locked `0.1.2`.
- [ ] `@luxledger/postgres-adapter`: installed/locked `0.1.3`.

## Contract and implementation drift

- [ ] Every registered public route is present in `openapi/openapi.yaml`, and every OpenAPI path is registered at runtime.
- [ ] Request/response schemas come from `@luxledger/http/contracts`; the demo does not duplicate transport DTOs.
- [ ] Status codes and bodies in README and `docs/SCENARIOS.md` match runtime responses.
- [ ] Auth descriptions match the 900-second TTL, roles, and per-request backing-key revocation check.
- [ ] Cursor limits, money strings, timestamp formats, error codes, rate limits, probes, and metrics match implementation.
- [ ] A fresh server returns byte-for-byte the checked-in spec from `GET /openapi.yaml`.
- [ ] The checked-in spec is byte-for-byte identical to `packages/http/openapi/openapi.yaml` in the reviewed LuxLedger source revision.
- [ ] Confirm LL-81 corrections remain effective in the released set:
  - ordinary transaction idempotency compares normalized nullable `description` and the complete entry multiset;
  - backdated posting documentation states that later materialized snapshots are updated transactionally;
  - domain errors crossing public export identities remain 4xx while raw database/unknown errors remain non-leaking 500 responses;
  - the demo contains no package-error workaround.

Useful checks:

```sh
npm run typecheck
curl -fsS http://localhost:3000/openapi.yaml > /tmp/luxledger-served-openapi.yaml
cmp openapi/openapi.yaml /tmp/luxledger-served-openapi.yaml
```

## Documentation and walkthrough

- [ ] Start from a clean clone or disposable worktree with no `.env`, `node_modules`, PostgreSQL volume, or bootstrap records.
- [ ] Follow README only: `npm ci`, key generation, `.env` loading, PostgreSQL, migration, bootstrap, API start.
- [ ] Confirm `/health` and `/ready` bodies/statuses.
- [ ] Run the entire README walkthrough and confirm create/retry/read/trial-balance outputs.
- [ ] Run every scenario in `docs/SCENARIOS.md`, including bulk rollback verification and history/reconciliation reads.
- [ ] Confirm a revoked backing API key makes its existing JWT fail on the next authenticated request.
- [ ] Validate all relative Markdown links resolve to files and all documented local endpoints exist.
- [ ] Review commands for POSIX-shell copy/paste behavior and ensure placeholders cannot be mistaken for literal IDs.
- [ ] Run `docker compose config --quiet`, `npm run typecheck`, and available smoke checks.

Simple local link check:

```sh
node -e 'const fs=require("fs"),path=require("path"); for(const f of ["README.md","ARCHITECTURE.md",...fs.readdirSync("docs").filter(x=>x.endsWith(".md")).map(x=>`docs/${x}`),"docs/releases/0.2.0.md"]){const s=fs.readFileSync(f,"utf8");for(const [,u] of s.matchAll(/\[[^\]]+\]\(([^)#]+)(?:#[^)]+)?\)/g)){if(!/^[a-z]+:/i.test(u)){const p=path.resolve(path.dirname(f),u);if(!fs.existsSync(p))throw Error(`${f}: missing ${u}`)}}}; console.log("Markdown file links OK")'
```

## PR hygiene

- [ ] Changes are scoped to LL-63 documentation/demo configuration; any discovered behavior defect is separately called out with a minimal fix and focused verification.
- [ ] No raw API key, JWT signing key, access token, tenant data, or local `.env` is committed.
- [ ] PR title references `LL-63`; package and OpenAPI versions are included in the PR description.
